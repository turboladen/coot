# billz-rqb.7 — Schema cache generation guard — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** A cache-clear (edit/delete/Refresh) concurrent with an in-flight `list_*` must not leave pre-edit schema cached — guard the cache write with a global generation counter.

**Architecture:** Add an `AtomicU64 generation` to `SchemaCache`; the clear ops (`invalidate`, `invalidate_connection`) bump it; `get_or_fetch` stamps the generation at entry and skips the write if it changed during `fetch().await` (return-but-don't-cache). One file: `core/src/schema.rs`.

**Tech Stack:** Rust (edition 2024), tokio tests.

## Global Constraints

- One file changes: `core/src/schema.rs`. No `mssql_client` in the public API; driver stays behind `core`.
- `get_or_fetch` must still NOT hold the `Mutex` across `.await` (the existing deadlock-avoidance) — the generation load/store are cheap atomic ops, no lock.
- `SchemaCache` stays `#[derive(Default)]` (`AtomicU64: Default` = 0).
- `cargo fmt` + `cargo clippy` clean (warnings = errors); `just verify` green.

**(c) `get_or_fetch`** — add the `generation` param + the write guard (replace the
whole fn at ~line 386):
```rust
async fn get_or_fetch<K, V, Fut>(
    map: &Mutex<HashMap<K, V>>,
    generation: &AtomicU64,
    key: K,
    fetch: impl FnOnce() -> Fut,
) -> Result<V>
where
    K: Eq + Hash + Clone,
    V: Clone,
    Fut: Future<Output = Result<V>>,
{
    let gen0 = generation.load(Ordering::Acquire); // stamp the epoch at entry
    if let Some(v) = map.lock().unwrap().get(&key) {
        return Ok(v.clone()); // guard dropped at the end of this `if`
    }
    let v = fetch().await; // no guard held across the await
    if let Ok(val) = &v {
        // rqb.7 (airtight): check the generation UNDER the insert lock, so
        // "observed gen0" and "inserted" are atomic w.r.t. this map. A clear
        // bumps the generation BEFORE clearing the maps, so either (a) its bump is
        // visible here and we skip, or (b) our insert precedes the clear's
        // map.lock().clear(), which then removes our entry. No stale row persists.
        // (The atomic load doesn't block; no `.await` is held across the guard.)
        let mut guard = map.lock().unwrap();
        if generation.load(Ordering::Acquire) == gen0 {
            guard.insert(key, val.clone());
        }
    }
    v // the in-flight caller still gets its value (return-but-don't-cache)
}
```

**(e) Bump on clear (BEFORE clearing — airtight ordering)** — in `invalidate` and
`invalidate_connection`, bump the generation *first*, then clear the maps. Bumping
before the clears is what makes the under-lock guard airtight: an insert that
observes the pre-bump generation is guaranteed to precede the clear's
`map.lock().clear()`, so the clear removes it.
```rust
    pub fn invalidate(&self) {
        self.generation.fetch_add(1, Ordering::Release); // rqb.7: bump BEFORE clearing
        self.databases.lock().unwrap().clear();
        self.tables.lock().unwrap().clear();
        self.views.lock().unwrap().clear();
        self.columns.lock().unwrap().clear();
    }
```
```rust
    pub fn invalidate_connection(&self, id: &ConnectionId) {
        self.generation.fetch_add(1, Ordering::Release); // rqb.7: bump BEFORE clearing
        self.databases.lock().unwrap().retain(|k, _| k != id);
        self.tables.lock().unwrap().retain(|k, _| &k.0 != id);
        self.views.lock().unwrap().retain(|k, _| &k.0 != id);
        self.columns.lock().unwrap().retain(|k, _| &k.0 != id);
    }
```
(`forget_connection` calls `invalidate_connection`, so it inherits the bump.)
