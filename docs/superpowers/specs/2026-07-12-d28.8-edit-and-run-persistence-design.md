# d28.8 — Edit-and-run persistence for opened saved queries

**Bead:** billz-d28.8 (P3, task) · **Date:** 2026-07-12 · **Status:** design approved

## Problem

Opening a saved query into a tab (d28.3) makes its SQL editable, but `run()`
persists remembered param values derived from the **edited** tab content against
`curSavedQuery`, whose stored `sql` is **never** written back. So editing a saved
query's SQL and running can leave the stored `sql` and `params` inconsistent:

- **Orphan params:** edit `@b` into the tab, Run → `@b`'s value is persisted onto
  the saved query even though its stored `sql` never references `@b`.
- **Dropped values:** that remembered `@b` is invisible on reopen —
  `deriveParams(storedSql, params)` scans the *stored* sql, which lacks `@b`, so
  `@b` is dropped from the bar (and purged on the next Run of the stored sql).

Non-crashing cruft; the common flow (open unchanged, fill, Run) is unaffected.
The bead asks to resolve the semantics and document them.

## Decision

**A saved query is a stable template, not a living document.** (Chosen over the
"Run = save edits" alternative, which would silently rewrite a curated library
entry on every Run — Run is a read action and must not redefine the library.)

Concretely: a saved query's persisted param set is defined by its **stored SQL**,
never by the edited tab content. On Run:

- **Declared params** (those in the stored SQL) are remembered as today.
- **Edited-in params** (new `@names` typed into the tab) and **SQL edits** are
  **scratch** — used for *this* run's execution, remembered nowhere.
- Editing a declared param *out* of the tab and running is **non-destructive**:
  its stored remembered value is preserved (a throwaway edit must not wipe a saved
  value).

`sql` was already never written back, so restricting the persisted params to the
declared set makes stored `sql` and stored `params` consistent **by
construction**. This matches how the app already works — Open loads a copy into a
scratch tab; Promote creates a *new* entry; there is no update-in-place.

## Design

### Pure logic (`paramBarLogic.ts`, unit-tested)

```ts
// d28.8: persist-back for an opened saved query under the "stable template" model.
// The saved query's param set is defined by its STORED sql (not the edited tab
// content): only DECLARED params are remembered; params edited into the tab are
// scratch (used for the run, discarded). A declared param not currently visible
// (edited out of the tab) keeps its stored value — editing is non-destructive.
export function persistDeclared(
  stored: SavedQuery,
  values: Record<string, string>,
): { params: Param[]; session: Record<string, string>; global: Record<string, string> } {
  const declared = deriveParams(stored.sql, stored.params);
  const toWrite = declared.filter((p) => p.name in values);
  const routed = routeWrites(toWrite, values);
  const written = new Map(routed.params.map((p) => [p.name, p]));
  const params = declared.map((p) => written.get(p.name) ?? p);
  return { params, session: routed.session, global: routed.global };
}
```

- `declared` = the canonical param set from the **stored** sql (merged with stored
  params, so scopes/types/values are retained).
- `toWrite` = declared params the user can currently see and set (`name in
  values`) — the param bar's `paramValues` is keyed by `curParams` (edited
  content), so a declared param edited out of the tab is absent from `values` and
  thus skipped (its stored entry survives via the `?? p` fallback).
- `routeWrites` (d28.4) is reused unchanged for tier routing (Local → `lastValue`;
  Session/Global → the returned maps + `lastValue` cleared).
- Edited-in params never appear in `declared`, so they reach neither `params` nor
  the session/global maps — scratch, as decided.

### `run()` change (`App.svelte`)

Replace the param persist block (currently `routeWrites(curParams, paramValues)`):

```ts
const routed = persistDeclared(curSavedQuery, paramValues);
await saveQuery({ ...curSavedQuery, params: routed.params }); // sql unchanged (already correct)
setSessionParams(routed.session);
setGlobalParams(routed.global);
```

The **execution path is unchanged**: `toResolvedParams(curParams, paramValues)` +
`runParams(id, effectiveDb, curTab.content, resolved)` still use the edited
content, so edited-in params run correctly this session. Only the *persistence*
narrows to the declared set. `setSessionParams`/`setGlobalParams` **merge** (not
replace), so narrowing the write maps leaves other params' shared values intact.

### What this makes explicit (and its complement)

SQL edits to an opened saved query never persist to the stored definition. The
deliberate counterpart — an explicit **"Update saved query"** action (the
non-silent version of the rejected "Run = save") — is filed as **billz-d28.10**
and is **out of scope here**.

## Testing

- **Pure (`persistDeclared`):**
  - orphan: edited-in `@b` present in `values` but not in stored sql → not in
    returned `params`, not in `session`/`global`.
  - normal fill: a declared Local param's value → persisted to its `lastValue`.
  - edit-out (non-destructive): a declared param absent from `values` keeps its
    stored `lastValue`/scope unchanged.
  - tier routing: a declared Session/Global param → value in the returned
    `session`/`global` map, `lastValue` cleared (delegates to `routeWrites`).
  - consistency: returned `params` names ⊆ `deriveParams(stored.sql)` names (never
    an orphan).
- **Gates + manual:** `just verify` green; manual smoke — open a saved query, edit
  in a `@b`, fill both, Run (rows return); reopen → only the stored-sql params
  show, no orphan `@b`; edit out a declared `@a`, Run, reopen → `@a`'s remembered
  value intact.

## Acceptance criteria

- After Run, an opened-then-edited saved query has stored `params` consistent with
  its stored `sql` (no orphan params; names ⊆ stored-sql params).
- Edited-in params execute for the run but are not remembered.
- Editing a declared param out of the tab and running preserves its stored value.
- Behavior documented (this spec + a code comment on `persistDeclared`/`run`).
- `cargo fmt`/`clippy` clean (no Rust change); `just verify` green; no
  `mssql_client` in `app`/`core` API (backend untouched).
