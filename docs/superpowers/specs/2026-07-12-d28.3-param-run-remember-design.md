# d28.3 — Parameterized run + remember-last-value (Local)

**Bead:** billz-d28.3 (P2, feature) · **Date:** 2026-07-12 · **Status:** design approved (staged)

## Problem

The parameterization core exists (bind vs raw-text substitution `d28.2`, `Param`
model `d28.1`, library UI `d28.6`) but there is **no end-to-end path to run a
parameterized saved query**: `core::run_with_params` is not wired to any Tauri
command, and the library's "Open" just dumps SQL into a scratch tab
(`newTabWithContent(q.sql)` — "does NOT run or bind params"). Saved queries also
have **no declared params** (`promoteToSavedQuery(…, null)` → `params: []`).

## Goal

Run a saved query with its `@param`s filled in, and **remember** the values so a
repeat run is one click. This is the first, Local-only slice of PLAN.md §5's
"remember-last-value is the feature."

## Scope

**In (d28.3):**
- Derive `@param`s from a saved query's SQL.
- A **param bar** in the editor tab: one row per param (value field + type
  indicator), a **Run** that binds/splices and executes.
- **Remember-last-value at Local scope**: entered values persist to
  `Param.last_value` (per saved query, via the existing `query_store`), so the
  bar pre-fills on reopen.
- **One-click rerun**: pre-filled fields ⇒ a repeat run is just clicking Run.
- Wire `core::run_with_params` to a Tauri command.

**Out (deferred to `d28.4`, the staged follow-up):**
- Session/Global value stores, cross-tier resolution precedence, and the
  per-param **scope selector**. This slice resolves a value from `last_value`
  only; `d28.4` generalizes `resolve()` to `local ?? session ?? global`.

**Out (other beads):** auto-type-from-catalog (`d28.5`); right-click-run-scoped
(`d28.7`).

## Design

### Param derivation (pure)
`deriveParams(sql: string, stored: Param[]): Param[]` — scan the SQL for
`@[A-Za-z_]\w*` tokens, dedup preserving first-appearance order, and **merge**
with `stored`: a name already in `stored` keeps its `sqlType`/`scope`/
`lastValue`; a newly-appeared name gets a default (`sqlType: null` = raw-text,
`scope: "local"`, `lastValue: null`). Lives in a rune-free `paramBarLogic.ts`,
unit-tested (mirrors `tabsLogic`/`savedQueriesLogic`).

*Accepted imperfection (single-user tool):* the regex can match `@x` inside a
string literal or comment, yielding a harmless extra bar field — NOT a safety
issue, because the actual substitution is still `d28.2`'s typed bind / flagged
raw-text splice. The deferred T-SQL lexer (`u2t`/§6) would tighten this later.
Documented on the function.

### Tab ↔ saved-query linkage
`QueryTab` gains `savedQueryId: string | null` (default `null`), persisted with
the tab and read-tolerant on deserialize (same backward-compatible pattern as
`database` in `cwt.9`). `newTabWithContent(content, database?, savedQueryId?)`
gains the third optional arg; `SavedQueryLibrary.openSavedQuery(q)` passes
`q.id` (and `q.targetDatabase`). Opening a saved query thus produces a tab that
knows which library item it came from.

### Run path
- New Tauri command `run_params(id, database, sql, params) -> QueryResult[]`
  delegating to `billz_core::run_with_params`. `params` is a `Vec<ResolvedParam>`
  (`{ name, sqlType, value }`) — core-owned types only.
- **Single batch** (no `GO`-splitting): the param path sends one batch, matching
  `run_with_params`'s existing contract. Documented; a parameterized query
  spanning `GO` batches is out of scope.

### Param bar (`ParamBar.svelte`, in `App.svelte`'s workspace)
Shown when the active tab has a `savedQueryId` **and** `deriveParams` yields ≥1
param; otherwise the normal toolbar Run (`run_sql`) applies unchanged (a
no-param saved query behaves like a scratch tab).

Per param row: a **value field** pre-filled with `param.lastValue ?? ""`; a type
indicator — the bind type (`int`, `nvarchar`, …) for `sqlType: Some`, or a
**loud `raw!` badge** for `sqlType: null` (§5: raw-text is injectable, render it
unsafe). The bar is **fields only — no Run button of its own**.

**One run control.** The existing toolbar **Run** (and Cmd/Ctrl-Enter) becomes
**param-aware** rather than adding a second button: `run()` in `App.svelte`
checks the active tab — if it has derived params, route to the parameterized
path; otherwise the current `run_sql` path is unchanged. This avoids two Run
buttons (and a toolbar Run that would fire `run_sql` on unbound `@param`s and
just fail). The parameterized path: read each field's current value → (1) update
the merged params' `lastValue` and persist the saved query via `save_query`
(remember — and this also *declares* newly-derived params permanently on first
run); (2) build `ResolvedParam[]`; (3) call `run_params`; (4) feed results into
the existing `results`/`messages`/`activeTab` state.

**One-click rerun** is emergent: the fields are pre-filled from `lastValue`, so
a repeat run is Run with no editing; a genuinely-unset param is simply an empty
field (no modal, no prompt).

## Testing
- **Headless (pure):** `deriveParams` — new names get defaults, existing keep
  metadata, order preserved, dups collapsed, `@x`-in-string caveat documented;
  building `ResolvedParam[]` from fields + params.
- **Rust:** `run_params` delegates to the already-tested `run_with_params`
  (env-gated live smoke that a bound `@p` returns rows).
- **Gates:** svelte-check + build for the bar; existing tab-serde tests updated
  for the new `savedQueryId` field (backward-compat).

## Acceptance criteria
- Opening a saved query with `@param`s shows a param bar; Run binds/splices and
  shows results in the grid.
- Entered values persist to `Param.last_value` and pre-fill on reopen.
- A fully-valued query reruns in one click (no prompt).
- Raw-text params (`sqlType: null`) render visibly unsafe.
- `cargo fmt`/`clippy` clean; `just verify` green; no `mssql_client` type in
  `app` or any `core` public API.
