# d28.5 — Param type selector + catalog-type mapping

**Bead:** billz-d28.5 (P3, feature) · **Date:** 2026-07-12 · **Status:** design approved

## Problem

d28.3/d28.4 shipped the param bar, but nothing sets `Param.sql_type`, so every
param is raw-text (spliced, flagged `raw!`) — the safe typed-bind path
(`sp_executesql`, built in d28.2) is unreachable from the UI. PLAN.md §5 wants
params "typed automatically from the schema, editable to override."

## Goal

Make a param a **typed bind** you can pick (and let the catalog interpreter that
`d28.7` will use to auto-fill from a column exist and be tested).

## Scope

**In:** a per-param **type selector** in the bar; immediate persistence of the
choice; a pure **catalog-type → `SqlType`** mapping (the "from the catalog"
interpreter).

**Out:** the auto-fill *trigger* (right-click table → scoped query with types
pulled from that table's columns) is **`d28.7`**, which consumes the mapping.
Decided in brainstorm: no query-text `FROM`-table guessing here (that needs the
deferred lexer); `d28.7` has the exact table, so auto-fill is precise there.

## Design

### Type selector (`ParamBar.svelte`)
Replace the static type chip with a `<select>` per param row: **raw-text**
(value `""` → `sqlType: null`) plus the nine `SqlType`s (`int`, `bigint`,
`nvarchar`, `bit`, `date`, `datetime2`, `decimal`, `uniqueidentifier`, `money`).
Bound to `p.sqlType` (or `""`); on change calls `onTypeChange(name, sqlType | null)`.
The loud **`raw!`** badge stays, rendered only when `sqlType` is `null` (selector
on "raw-text") — so an unsafe raw-text param is still visibly flagged. The scope
selector + inherited badge from d28.4 are unchanged.

### Persist on change (`App.svelte`)
`onTypeChange(name, sqlType)` persists immediately via `save_query` — rebuild
`curSavedQuery.params` with that param's `sqlType` updated (declaring a
newly-derived param) — mirroring d28.4's `onScopeChange`. The value still writes
on Run (unchanged). A newly-typed param routes through d28.2's `sp_executesql`
bind path on the next Run.

### Catalog-type → `SqlType` mapping (`paramBarLogic.ts`, pure)
`catalogTypeToSqlType(dataType: string): SqlType | null` — take the base type
from a `ColumnInfo.data_type` canonical string (strip any `(...)`:
`"nvarchar(50)"`→`nvarchar`, `"decimal(19,4)"`→`decimal`, `"int"`→`int`), lowercase,
and map into the capped `SqlType` set:
- `int` / `tinyint` / `smallint` → `"int"`; `bigint` → `"bigint"`
- `nvarchar` / `nchar` / `varchar` / `char` → `"nvarchar"`
- `bit` → `"bit"`; `date` → `"date"`
- `datetime2` / `datetime` / `smalldatetime` → `"datetime2"`
- `decimal` / `numeric` → `"decimal"`; `money` / `smallmoney` → `"money"`
- `uniqueidentifier` → `"uniqueidentifier"`
- anything else (`float`, `real`, `time`, `binary`, `varbinary`, `xml`, …) → `null` (raw-text)

**Widening is intentional** (`smallint`→`int`, `datetime`→`datetime2`,
`varchar`→`nvarchar`): `SqlType` is the deliberately capped "things you filter by"
set, and the value binds correctly because the driver derives the `sp_executesql`
type declaration from the bound value at runtime (query.rs note / d28.2), not from
this tag. This is the interpreter `d28.7` will call with a right-clicked column's
`data_type`; it is exported and tested now to de-risk `d28.7` (unused in d28.5's
own runtime — no auto-fill trigger here yet).

## Testing
- **Pure:** `catalogTypeToSqlType` matrix — each family maps correctly; width
  suffixes stripped (`nvarchar(50)`/`decimal(19,4)`); widened types
  (`smallint`→int, `datetime`→datetime2, `varchar`→nvarchar); unmappable
  (`float`, `time`, `xml`) → null; case-insensitive.
- **Gates:** svelte-check + build for the selector; manual smoke — pick `int` for
  a param → `raw!` badge clears; Run binds via `sp_executesql` (typed), not a
  raw splice.

## Acceptance criteria
- A param's type is settable in the bar (raw-text + the 9 SqlTypes); the choice
  persists (survives reopen) and a typed param runs as a safe bind.
- `raw!` shows only for raw-text (`sqlType: null`) params.
- `catalogTypeToSqlType` maps catalog canonical types into `SqlType` (widening as
  specified) or `null`, and is exported for `d28.7`.
- `cargo fmt`/`clippy` clean (no Rust change expected); `just verify` green; no
  `mssql_client` in `app`/`core` API (backend untouched).
