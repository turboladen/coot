# billz-rqb.7 ŌĆö Schema cache generation guard (no stale re-cache mid-edit)

**Bead:** billz-rqb.7 (P4, bug) ┬Ę **Date:** 2026-07-12 ┬Ę **Status:** design approved

## Problem

`SchemaCache::get_or_fetch` (schema.rs) releases its `Mutex` across `fetch().await`
(deliberately ŌĆö holding it across the await would deadlock/serialize), then
unconditionally inserts a successful result:

```rust
if let Some(v) = map.lock().unwrap().get(&key) { return Ok(v.clone()); } // hit
let v = fetch().await;                                                   // no lock held
if let Ok(val) = &v { map.lock().unwrap().insert(key, val.clone()); }    // ŌåÉ races
v
```

If a connection is edited/deleted while a `list_*` is in flight,
`forget_connection` clears that connection's cache entries and evicts the warm
session ŌĆö but the in-flight `fetch` (still querying the **old** server) returns
afterward and re-inserts the pre-edit rows into the just-cleared cache. The tree
then shows stale schema until a manual Refresh. Rare (requires editing a
connection's server/creds while a subtree expand is mid-flight), single-user, but
a real correctness gap surfaced by the billz-lpb review.

## Goal

Editing/refreshing while an expand is in flight must not leave pre-edit schema
cached. The stale in-flight result is returned to its one caller but **not
cached**.

## Design

**A global generation counter** stamped before each fetch; the cache write is
skipped if the generation changed during the fetch (i.e. a clear happened).

### `SchemaCache` gains a generation

```rust
pub struct SchemaCache {
    databases: Mutex<HashMap<ConnectionId, Vec<DatabaseInfo>>>,
    // ŌĆ” tables / views / columns ŌĆ”
    sessions: SessionCache,
    /// Bumped by every cache-clearing op; `get_or_fetch` refuses to write a
    /// result fetched across a bump (rqb.7). One global counter (not per-
    /// connection) ŌĆö simplest with the generic helper; a cross-connection false
    /// invalidation just costs a harmless re-fetch on a single-user tool.
    generation: std::sync::atomic::AtomicU64,
}
```

`AtomicU64` is `Send + Sync` and needs no `derive` change (`SchemaCache: Default`
still holds ŌĆö `AtomicU64: Default` = 0). `AtomicU64` is not `Clone`, but
`SchemaCache` isn't `Clone` (it lives behind Tauri's managed state by reference),
so that's fine.

### The clear ops bump the generation

`invalidate` (Refresh, clears all) and `invalidate_connection` (retains all but
one connection) each `self.generation.fetch_add(1, Ordering::Release)` **before**
clearing the maps. `forget_connection` already calls `invalidate_connection`, so
it inherits the bump. Any cache removal thus advances the epoch ŌĆö and bumping
*before* the clears is what makes the under-lock guard airtight (below).

### `get_or_fetch` becomes generation-aware

```rust
async fn get_or_fetch<K, V, Fut>(
    map: &Mutex<HashMap<K, V>>,
    generation: &AtomicU64,
    key: K,
    fetch: impl FnOnce() -> Fut,
) -> Result<V>
where K: Eq + Hash + Clone, V: Clone, Fut: Future<Output = Result<V>> {
    let gen0 = generation.load(Ordering::Acquire);           // stamp the epoch
    if let Some(v) = map.lock().unwrap().get(&key) { return Ok(v.clone()); }
    let v = fetch().await;                                   // no lock held
    if let Ok(val) = &v {
        // rqb.7 (airtight): check the generation UNDER the insert lock so
        // "observed gen0" and "inserted" are atomic w.r.t. this map. Clears bump
        // BEFORE clearing, so an insert that sees gen0 precedes the clear's
        // map.lock().clear(), which removes it; a visible bump makes us skip.
        let mut guard = map.lock().unwrap();
        if generation.load(Ordering::Acquire) == gen0 {
            guard.insert(key, val.clone());
        }
    }
    v // the in-flight caller still gets its value (return-but-don't-cache)
}
```

The four callers (`databases`/`tables`/`views`/`columns`) pass
`&self.generation` as the new second arg.

### Ordering & the airtight guarantee

`gen0` is captured at function entry; the post-fetch comparison happens **under the
target map's lock**, and clears **bump before clearing**. That combination is
airtight: if a concurrent clear's bump is visible when we hold the insert lock, we
skip; if it isn't, our insert completes before the clear can acquire the lock to
clear that map, so the clear removes our (stale) entry. Either way no stale row
survives ŌĆö closing even the sub-microsecond window between the gen-load and the
insert that a plain "load, then separately lock+insert" would leave on a
multi-threaded runtime.

`Acquire`/`Release` pair the load against the clear ops' `fetch_add`
(`fetch_add` always increments, so two clears net `+2` ŌĆö never an ABA back to
`gen0`). On a cache **hit** the generation is irrelevant (a prior clear would have
removed the entry). Only the fetch-and-insert path is guarded.

## Testing

Deterministic unit tests drive `get_or_fetch` directly (no DB, no timing):
- **`get_or_fetch_skips_cache_when_generation_bumped_during_fetch`** ŌĆö the fetch
  closure calls `generation.fetch_add(1, ŌĆ”)` mid-`await`; assert the caller gets
  the value AND the map has no entry for the key (stale-guard holds).
- **`get_or_fetch_caches_when_generation_stable`** ŌĆö no bump; assert the value is
  cached.
- **`get_or_fetch_returns_cached_without_fetch`** ŌĆö pre-populated map returns the
  cached value without invoking `fetch` (existing behavior preserved).
- **`invalidate_and_invalidate_connection_bump_generation`** ŌĆö a `#[cfg(test)]`
  accessor (or `Ordering::Relaxed` load in the test) confirms each clear op
  advances the counter.

Live env-gated smoke tests are unchanged (the generation is transparent to them).

## Acceptance criteria

- A cache-clear (`invalidate` / `invalidate_connection` / `forget_connection`)
  concurrent with an in-flight fetch cannot leave the fetched rows cached.
- The in-flight caller still receives its (uncached) result; a subsequent
  expand/Refresh re-fetches fresh.
- Normal (no concurrent clear) fetches cache exactly as before; cache hits are
  unchanged.
- `cargo fmt`/`clippy` clean; `just verify` green; no `mssql_client` in
  `app`/`core` public API.
