# d28.10 — Explicit "Update saved query" action

**Bead:** billz-d28.10 (P3, feature) · **Date:** 2026-07-12 · **Status:** design approved

## Problem

d28.8 decided a saved query is a stable template: Run never persists SQL edits
back to the definition (edits are scratch). That leaves **no way to update an
existing saved query's SQL in place** — "Promote current tab" only creates a
*new* entry (new UUID). Editing an opened saved query and wanting to keep the
change is a dead end.

## Goal

Give the deliberate, explicit counterpart to the rejected "Run = save" option: a
one-click **Update saved query** that writes the tab's edited SQL (and reconciled
param declarations) back to the linked saved query.

**Model:** completes d28.8's pair — **Run executes (never redefines); Update
redefines (explicit only).**

## Scope

**In:** a toolbar button next to Run, shown only for a tab linked to a saved
query, enabled only when the tab's SQL has diverged from the stored definition;
on click it saves `sql` + reconciled `params` back to that saved query.

**Out:** capturing the tab's target database (stored `targetDatabase` is left
untouched — decided in brainstorm); a Cmd-S keyboard shortcut; a dirty-dot on the
tab title (the button's enabled state already signals unsaved edits); any change
to Run / param-value persistence (that stays d28.8's tier routing).

## Design

### Placement & gating (`App.svelte` toolbar)

A button after the Run button, rendered only when the active tab is linked to a
saved query (`curSavedQuery` non-null), and **enabled only when diverged**:

```svelte
{#if curSavedQuery}
  <button
    onclick={updateSavedQuery}
    disabled={!dirty}
    title="Save the tab's edited SQL back to this saved query"
  >
    Update saved query
  </button>
{/if}
```

```ts
const dirty = $derived(!!curSavedQuery && !!curTab && curTab.content !== curSavedQuery.sql);
```

The disabled↔enabled flip is the "unsaved edits" signal. After a successful
Update, `saveQuery` refreshes the library (`savedQueries.svelte` `save` awaits
`refresh`), so `curSavedQuery.sql === curTab.content` → `dirty` goes false → the
button disables again (implicit "saved" feedback). **No modal confirm** — an
explicit click on an enabled button is the intent, and (unlike Delete) the change
is recoverable by re-editing.

### The write

```ts
async function updateSavedQuery() {
  if (!curSavedQuery || !curTab || !dirty) return;
  await saveQuery({
    ...curSavedQuery,
    sql: curTab.content,
    params: deriveParams(curTab.content, curSavedQuery.params),
  });
}
```

`deriveParams(newSql, curSavedQuery.params)` reconciles the declared param set to
the edited SQL (reusing existing d28.3 logic): a surviving param keeps its
`sqlType`/`scope`/`lastValue`; a param edited *in* becomes a real declared param
(so later Runs remember it, per d28.8); a removed param drops. `targetDatabase`
and the param **values** (`paramValues`) are untouched — value persistence stays
with Run's tier routing.

After the write, the existing reactivity settles cleanly: `curSavedQuery`
recomputes from the refreshed `library.list`; `curParams` re-derives from the new
stored sql (equal to `curTab.content`); the `paramValues` `$effect` runs a
same-tab recompute (preserves typed values, seeds any newly-declared param).

### Why no new pure logic

The feature is a `$derived` (`dirty`) + an async handler (`updateSavedQuery`) + a
button, all in `App.svelte`, composing the existing `deriveParams` and
`saveQuery`. Nothing new is worth extracting or unit-testing; verification is
gates + a manual smoke.

### Edge notes

- **Exact-string divergence** (`!==`): a trailing-newline-only difference reads as
  dirty. Honest and simplest; acceptable for a single-user tool.
- **Guard**: `updateSavedQuery` re-checks `dirty` (defends against a stale click if
  the button somehow fires while clean); a no-op otherwise.

## Testing

- **Gates:** `just ui-check` (svelte-check 0 errors), `just ui-test`, `just
  ui-build`.
- **Manual smoke:** Open a saved query → Update is disabled (not diverged). Edit
  the SQL (e.g. add `AND x=@x`) → Update enables. Click → button disables again;
  reopen the query → the edited SQL + the new `@x` param persist. Add a param,
  Update, Run → `@x` is remembered (now a declared param). A tab NOT linked to a
  saved query shows no Update button.

## Acceptance criteria

- A tab linked to a saved query shows an **Update saved query** button, enabled
  only when its SQL differs from the stored definition; a plain scratch tab shows
  none.
- Clicking it persists the tab's SQL + reconciled param declarations to that saved
  query (distinct from Promote, which creates a new entry); reopening reflects the
  update, and the button returns to disabled.
- `targetDatabase` and param values are unchanged by Update.
- `cargo fmt`/`clippy` clean (no Rust change); `just verify` green; no
  `mssql_client` in `app`/`core` API (backend untouched).

## Files

- Modify: `app/ui/src/App.svelte` — `dirty` derived, `updateSavedQuery` handler,
  the toolbar button. (No new module, no Rust, no pure-logic test file.)
