# d28.4 — Param scope tiers (Global < Session < Local)

**Bead:** billz-d28.4 (P3, feature) · **Date:** 2026-07-12 · **Status:** design approved

## Problem

d28.3 shipped the param bar + one-click rerun with **Local-only** remembered
values (`Param.last_value` per query). PLAN.md §5 wants three tiers: a param
value can be remembered per-query (Local), for the whole session (Session — "set
`@cust` for the afternoon", shared across every query using `@cust`), or as a
persisted default (Global — e.g. `@today`). Resolution layers them.

## Goal

Add Session and Global value tiers plus a per-param **scope selector**, so a
value resolves `Local ?? Session ?? Global` and edits are remembered into the
param's chosen tier.

## Scope

**In:** Session + Global stores; the resolve-precedence chain; the scope selector
in the param bar; write-routing on Run; the **inherited-value badge** (chosen
mockup option B).

**Out:** auto-type from catalog (`d28.5`); right-click-run-scoped (`d28.7`).

## Design

### Decisions carried from the d28.3 brainstorm
- **Resolution = precedence + write-target** (confirmed): `resolve(@name) =
  Local ?? Session[name] ?? Global[name]`. The scope selector picks where a typed
  value is *remembered*, independent of where the displayed value resolved from.
- **Write timing = on Run** (confirmed): consistent with d28.3's Local remember.
- **Global persists to localStorage**; Session is in-memory.

### Stores (new, name-keyed by `@name`)
- `app/ui/src/lib/sessionParams.svelte.ts` — `export const sessionParams =
  $state<Record<string,string>>({})`, in-memory (app-session lifetime), plus a
  `setSessionParams(writes: Record<string,string>)` that merges.
- `app/ui/src/lib/globalParams.svelte.ts` — same shape, persisted to
  **localStorage** under a versioned key, mirroring `tabs.svelte.ts` (load on
  module init with a try/catch that degrades to `{}`; `setGlobalParams(writes)`
  merges + persists).
- **Local** stays `Param.last_value` in `query_store` — unchanged.

### Pure logic (`paramBarLogic.ts`, unit-tested)
- `resolve(param, session, global): string | null` =
  `param.lastValue ?? session[param.name] ?? global[param.name] ?? null`.
- `valueSource(param, session, global): "local" | "session" | "global" | null`
  — where the resolved value comes from (`local` if `lastValue != null`, else
  `session` if `session[name] != null`, else `global` if `global[name] != null`,
  else `null`). Drives the inherited badge.
- `routeWrites(params, values): { params: Param[]; session: Record<string,string>; global: Record<string,string> }`
  — on Run, route each param's field value to **its scope**:
  - **Local** → returned param has `lastValue = value`.
  - **Session** → `session[name] = value` **and** returned param has
    `lastValue = null` (so `resolve` falls through to the Session value).
  - **Global** → `global[name] = value` **and** `lastValue = null`.
  Params keep their `scope`. Returned `session`/`global` maps are merged into the
  stores by the caller.
- `nextParamValues(...)` (from d28.3) is extended: pre-fill uses `resolve(...)`
  instead of `param.lastValue`. It reads the Session/Global stores as **untracked
  snapshots** in the App effect, so a Session value set elsewhere propagates to a
  tab on its next switch/open — not disruptively into a field mid-edit.

### Param bar (`ParamBar.svelte`)
Each row gains, after the type chip:
- a **scope `<select>`** (Local / Session / Global) bound to the param's scope;
  changing it calls an `onScopeChange(name, scope)` prop.
- an **inherited badge** (option B) shown only when `valueSource` is `session`
  or `global`: `↳ session` (green) / `↳ global` (blue). A locally-set value
  (`local`) or unset (`null`) shows no badge. `title` explains the tier.

### App wiring (`App.svelte`)
- Import the two stores + `resolve`/`valueSource`/`routeWrites`.
- `onScopeChange(name, scope)` persists immediately: rebuild `curSavedQuery.params`
  with that param's `scope` updated (declaring it if newly-derived) and
  `save_query`. (Scope is a rare, deliberate config change — immediate persist
  avoids an extra state map; value still writes on Run.)
- Compute a per-param source map (`valueSource` over `curParams` + the store
  snapshots) to pass to `ParamBar` for the badges.
- Run (param branch): `const { params, session, global } = routeWrites(curParams,
  paramValues)` → `save_query({ ...curSavedQuery, params })` + `setSessionParams
  (session)` + `setGlobalParams(global)` → build `ResolvedParam[]` (unchanged) →
  `run_params`.

### The inherited-value subtlety (accepted, option B)
`valueSource` is computed from *stored* state (`lastValue`/stores), so a field
the user has typed into but not yet Run still shows an inherited badge until the
Run persists it. Acceptable for a single-user tool — the badge means "no Local
value stored yet; inheriting." Documented on `valueSource`.

## Testing
- **Pure:** `resolve` precedence (all four fall-through cases); `valueSource`
  (local/session/global/null); `routeWrites` (Local→lastValue; Session/Global→
  store + lastValue cleared; scope preserved); `nextParamValues` pre-filling from
  `resolve` across tiers.
- **Stores:** `globalParams` localStorage round-trip + degrade-on-corrupt (mirror
  the tabs tests); `sessionParams` merge.
- **Gates:** svelte-check + build for the scope selector + badge; manual smoke
  (set `@cust` Session in one query, open another using `@cust`, see it inherited
  with the badge; Global default; Local override).

## Acceptance criteria
- A param's value resolves `Local ?? Session ?? Global`; the scope selector sets
  where an edit is remembered; setting a value Session-scoped in one query makes
  every query referencing that `@name` resolve to it this session.
- Global values persist across app restart (localStorage); Session values do not.
- Inherited values show a `↳ session` / `↳ global` badge; locally-set values do not.
- `cargo fmt`/`clippy` clean; `just verify` green; no `mssql_client` in `app`/`core` API.
