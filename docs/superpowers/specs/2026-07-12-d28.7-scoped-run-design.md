# d28.7 — Right-click table → run saved query scoped to it

**Bead:** billz-d28.7 (P2, feature) · **Date:** 2026-07-12 · **Status:** design approved

## Problem

The tree, the saved-query library, and the param machinery (d28.3–d28.5) all
exist but don't connect. PLAN.md §7's payoff — "the tree and the query library
sharing one data model" — is unbuilt: you can't take a saved query and point it
at a table you're looking at.

## Goal

Right-click a table in the object tree → run a saved query "scoped" to it:
`@table` pre-filled from the node, and params matching that table's columns
auto-typed via d28.5's `catalogTypeToSqlType`. Closes the Phase-3 epic.

## Scope

**In:** a tree context menu on table nodes; a `Run saved query ▸` submenu
filtered to queries that reference `@table`; scoped-open that reuses the entire
d28.3–d28.5 tab/param-bar/run flow (no new run path).

**Out:** no new run mechanism; no fuzzy param↔column matching (name-equality
only); double-click → `SELECT TOP 1000` (rqb.6) is unchanged.

## Design

### Context menu (`TableNode.svelte`)
Right-clicking a table node (`oncontextmenu`, `preventDefault`) opens a menu — a
`position: fixed` `<div>` at the cursor (fixed so the tree's `overflow:auto`
can't clip it), dismissed on outside-click / `Esc`. It contains **`Run saved
query ▸`** with a submenu listing only saved queries whose SQL references
`@table` (a literal `@table`-substring scan of `library.list`). When none
qualify, the item is disabled with a hint. The existing double-click →
`SELECT TOP 1000` stays.

### Scoped-open (reuses d28.3–d28.5 — no new run path)
Picking a query from the submenu:
1. Fetch the clicked table's columns (`list_columns` via `SchemaCache`).
2. `deriveParams(query.sql, query.params)` → the param list; `autoTypeParams(...)`
   fills each **unset (`sqlType: null`)** param whose name (minus `@`,
   case-insensitive) matches a column with `catalogTypeToSqlType(column.data_type)`
   — **`@table` is excluded** (it's the table identifier, stays raw-text), and
   already-typed params are left alone (manual overrides win).
3. Set the `@table` param's `last_value` to the node's bracket-quoted
   `[schema].[table]` (reusing `quoteIdent` from `selectTopQuery.ts`).
4. `save_query` the query with the auto-typed params + `@table`'s value.
5. Open it in a new tab linked to the saved query (`newTabWithContent(sql,
   targetDatabase, savedQueryId)`), making it active.

The existing param-bar reactivity then does the rest: the `paramValues` effect
resolves each field from `resolve()` — `@table` shows the node ref (from its
`last_value`), column params show their types, the bar renders. **Not auto-run** —
you fill any remaining values and hit Run (which routes through the d28.3/d28.5
`run_params` path). This is the elegance: scoped-open is *just* a `save_query` +
open; nothing new in the run path.

### Why persist on open (not a transient seed)
Setting `@table`'s `last_value` + auto-types via `save_query` means the existing
`resolve()`/effect populate the bar with no special seeding hook. `@table`'s
value is per-invocation by nature (each right-click overwrites it with the new
table — "last table scoped" — which is the desired behaviour). Auto-typed types
persist so the query stays typed.

### Placement of the scoped-open logic
`TableNode` already imports `newTabWithContent`; it additionally uses `library`
(submenu filter), `saveQuery` (savedQueries store), `listColumns` (api), and the
pure helpers. To keep `TableNode` focused, the async orchestration goes in a small
helper `openScopedQuery(connId, db, schema, table, savedQuery)` (its own module),
called from the menu; App's param bar reacts to the newly-active tab (no callback
prop-drilling through the tree).

## Pure logic (`paramBarLogic.ts`, unit-tested)
- `queriesReferencingTable(queries: SavedQuery[]): SavedQuery[]` — those whose
  `sql` contains the literal `@table` token (word-boundary; a `@table2`
  false-positive just lists an extra query — harmless).
- `autoTypeParams(params: Param[], columns: ColumnInfo[]): Param[]` — return
  `params` with each param that (a) isn't `@table`, (b) has `sqlType: null`, and
  (c) name-matches a column (case-insensitive, `@` stripped) set to
  `catalogTypeToSqlType(col.data_type)` (which may itself be `null` for a
  float/real/etc. column → stays raw-text). Non-matching / already-typed / `@table`
  params unchanged.

## Testing
- **Pure:** `queriesReferencingTable` (includes `@table` refs, excludes others,
  `@table2` edge); `autoTypeParams` (fills unset matching, skips `@table`, skips
  already-typed, unmappable column type → stays null, name case-insensitive).
- **Gates + manual:** context menu + scoped-open wiring via svelte-check/build;
  manual smoke — right-click a table → the submenu lists only `@table` queries →
  pick one → tab opens with `@table` = `[dbo].[Orders]`, a `@cust` param
  auto-typed `int`, param bar shown → fill + Run returns rows.

## Acceptance criteria
- Right-clicking a table shows a `Run saved query ▸` submenu of only the saved
  queries referencing `@table` (disabled when none).
- Picking one opens it in a tab with `@table` set to the node's `[schema].[table]`
  and column-matching unset params auto-typed; the param bar shows; it does not
  auto-run.
- Auto-type never overwrites a manually-set param type; `@table` stays raw-text.
- `cargo fmt`/`clippy` clean; `just verify` green; no `mssql_client` in app/core API.
